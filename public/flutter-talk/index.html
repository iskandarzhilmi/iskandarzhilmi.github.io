Tool 01

RevenueCat

TRANSITION (memorized): "An app isn't a business until it can get paid. So tool one." · vs DIY: the purchase is 1 line either way, the other 10,000 lines are the hard part · server-side receipt validation (else jailbreak = free premium) · lifecycle edge cases: renewals, billing retry, grace periods, refunds, proration, restore · StoreKit 1/2 + Play Billing forced migrations, RC absorbs them · cross-device sync with no auth · analytics + remote offerings free · that's the 1% fee

RevenueCat

In-app purchases

  • Subscriptions & one-time purchases (points)
  • No backend required (it's optional)
Subs + consumable points · no backend needed · TALKING POINT: my app has NO auth, no login · why: frictionless, less friction → higher conversion · anonymous IDs, tied to store account · every signup screen = drop-off

4 terms to know

Example: a workout app

Product · Package · Offering · Entitlement

1 fake workout app → 4 RC terms · these confused me at first, every tutorial assumes them · mental model: Product = what you sell, Package = cross-store wrapper, Offering = the paywall lineup, Entitlement = what it unlocks · flow: buy Product → unlocks Entitlement → app checks it

Product

Each item you sell in the store.

gym_weekly_199$1.99 / week
gym_monthly_499$4.99 / month
gym_yearly_3999$39.99 / year
gym_lifetime_9999$99.99 once
Products = real SKUs in App Store Connect / Play Console, mirrored in RC dashboard · subs, consumables, lifetime: all products · ID convention: name_period_price · price in ID because price change usually = new product · the only part the stores see

Package

One product, grouped across both stores.

$rc_weekly gym_weekly_199 gym_weekly
$rc_monthly gym_monthly_499 gym_monthly
$rc_annual gym_yearly_3999 gym_yearly
Same product wrapped across both stores · code asks $rc_monthly, no platform branching · $rc_weekly / $rc_monthly / $rc_annual are RC's default identifiers, best practice over custom ones · iOS + Android product IDs differ, package hides that · 1 code path

Offering

The set of packages a paywall shows.

default
Weekly$1.99
Annual$39.99
Lifetime$99.99
black_friday
Annual −40%$23.99
Lifetime$59.99

Offering = the set of packages a paywall shows · app calls getOfferings().current → whatever the dashboard says · swap lineup remotely, no app update · Black Friday: switch current offering, switch back after · win-backs, A/B price tests · best practice: never hardcode product IDs in the paywall · can mix subs + consumables in one offering

Entitlement

Many products unlock the same access.

$rc_weekly$rc_monthly$rc_annual$rc_lifetime
→
premium
entitlement
→
Unlimited workouts
what they get
Entitlement = the access level a purchase unlocks · best practice: check entitlement, never product ID · 1 boolean: has "premium"? · which of the 4 products = irrelevant · add/retire products later, app code unchanged · most apps need just 1 entitlement, tiered apps use several · gotcha: consumables in an entitlement unlock it forever

Example

Locking a premium feature

final info = await Purchases.getCustomerInfo();
final isPremium =
    info.entitlements.active.containsKey("premium");

if (isPremium) {
  openWorkoutPlan();
} else {
  showPaywall();
}
The whole point of entitlements in 1 snippet · premium feature → check flag → open it or show paywall · no product IDs in app code · works for all 4 products + any added later

Error tracking for real-world apps

Sentry

Why Sentry?

  • Real error logs from real devices
  • Catch crashes you'd never see
Logs from real phones · crashes after ship · TALKING POINT: lifesaver on Android camera quirks, every manufacturer's camera behaves differently · Andrea: url_launcher intent broke links, caught day 1

Example

The setup

await SentryFlutter.init(
  (options) => options.dsn = 'https://...@sentry.io/...',
  appRunner: () => runApp(const MyApp()),
);
This is the entire setup: wrap runApp once · every uncaught Dart + native crash auto-captured from then on · no try/catch scattered everywhere · 5 minutes from pub add to first event · (if asked: also does session replay, tracing, release health, I don't use them)

Tool 03

PostHog

Product analytics & more

TRANSITION (memorized): "RevenueCat tells me about money, Sentry about errors. Neither tells me what users actually DO." · before: guessing, gut feeling vs measuring

PostHog

Product analytics

  • Track events: scans, paywall views, purchases
  • Funnels: see exactly where users drop off
  • Session replay: watch real user sessions
Track scan_started, paywall_viewed, purchase · funnel showed which onboarding step leaked · free: 1M events + 5K replays/mo

Example

Tracking an event

await Posthog().capture(
  eventName: 'scan completed',
  properties: {'method': 'barcode', 'result': 'halal'},
);
1 call per business event · naming convention: object + verb ("scan completed") · screens come free via PosthogObserver autocapture, manual capture = business events only · these events are what funnels are built from

PostHog

Beyond analytics

  • Feature flags: roll out or kill a feature remotely, no release
  • Experiments: A/B test a change, measure the impact
  • Surveys: ask users in-app, no code

All in one SDK — autocapture, screen tracking, replay.

Flags: remote toggle, no release · experiments + stats · in-app surveys (Flutter: surveys=true + PosthogObserver, then zero code per survey, create in dashboard) · autocapture + PosthogObserver screen tracking

PostHog MCP

MCP: Model Context Protocol

in Claude Code

Claude Code = Anthropic's AI agent that lives in your terminal · MCP (Model Context Protocol) = open standard that plugs external tools into the AI, PostHog ships an MCP server · result: the AI can query my real analytics directly · how it helps: ask in plain English, it writes + runs the queries, finds the insights (biggest leak, slowest step) on its own · analytics becomes a conversation, no SQL, no dashboard clicking · this funnel = one sentence prompt

Tool 04

Shorebird

Push Flutter code updates over the air

TRANSITION (memorized): "PostHog finds the leak. How fast can you ship the fix?" · OTA Dart patches, skip the review queue

Shorebird

Why Shorebird?

  • Push updates without a new build
  • Bypasses app store submission review
  • Great for small UX improvements, no store update
shorebird release replaces flutter build · shorebird patch = OTA Dart · copy tweaks, layout fixes live same day

Shorebird

Pricing

PlanIncludesPrice
Free5K patch installs/month$0
Pro50K patch installs/month$20/mo
Business1M patch installs/month$400/mo
Billed per successful patch install, not per release · free covers 5K installs/mo, plenty for a small app · overage $1 per 2,500 installs

Shorebird

Limitations

  • Patch downloads on launch, applies on the next restart
  • Dart only: assets, native code & engine changes can't be patched
  • Not suitable for critical bug fixes
2 launches: 1st downloads, 2nd runs · broken user stays broken until restart ×2 · Dart only: no assets / native / engine · polish, not emergencies

Critical bug?
Everyone must update.

Shorebird = small fixes · critical bug = real store release · but users don't update for weeks → old broken versions linger · answer = tool 5

Tool 05

force_update_helper

Prompt users to update when a new version ships

by Andrea Bizzotto (the Flutter legend!)

Andrea Bizzotto, codewithandrea.com · min-version check + blocking dialog · minutes to wire up

force_update_helper

Why force updates?

  • Fatal errors & DTO mismatches with the backend happen
  • Old versions in the wild keep breaking silently
  • Some users turn auto-updates off entirely
  • Host the minimum version on a GitHub Gist or backend
  • I prefer Gist: less backend-dependent, easier solo
Fatal errors, DTO mismatches · old versions break silently · auto-update is optional: iOS Settings → App Store toggle, Android global or per-app, those users sit on old versions forever with a worse experience · Gist > backend: simpler solo · team: agree gist owner

Example

force_update_config.json

{
  "config": {
    "minimum_version": "1.4.6"
  }
}

Hosted on a GitHub Gist

Whole "backend" = JSON on a Gist, fetched on start · bump min_version → old versions block · free kill switch

Tool 06

in_app_review

Ask for ratings with the native prompt, in-app

by Britannio Jarrett

TRANSITION (memorized): "Users are happy now. Last tool makes them say it in public." · wraps SKStoreReviewController (iOS) + Play In-App Review (Android) · system sheet, never leaves app

in_app_review

Asking for ratings

  • Only angry users review unprompted
  • Important for ASO (App Store Optimization)
  • Ask at the right moment (e.g. after the first successful scan)
  • Don't abuse it (iOS limits the prompt to 3× per year)
Why do big apps + government apps sit at 3.2★? They never ask for reviews · so the only reviewers left = angry users who go out of their way to the App Store · happy users stay silent unless prompted at a happy moment · hurts the look, and for a B2C / solo app the look IS the funnel: rating count + average → ASO ranking + install conversion · iOS: 3 per 365 days, auto-enforced, request ≠ guarantee · Andrea: likes/engaged user picked the moment, exponential backoff

The native rating prompt

Fire right after a win: first successful scan
